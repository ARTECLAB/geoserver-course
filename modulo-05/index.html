Seguridad &
Control de Acceso 🔒

No todos deben ver todo. Usuarios, roles, restricción por capa y workspace. Protege tu GeoServer como un profesional.

📅 Clase 6 ⏱️ 2 horas 🎯 Intermedio

🎯 Objetivo

Configurar usuarios y roles en GeoServer, restringir acceso a capas y servicios por usuario, y aplicar reglas de seguridad a nivel de workspace, capa y operación.

🛡️ ¿Por qué necesitas seguridad en GeoServer?

El problema real

Hasta ahora, tu GeoServer está abierto: cualquiera que conozca la URL puede ver todas las capas, descargar todos los datos y acceder a la administración con admin/geoserver. En un proyecto real esto es inaceptable.

Escenarios reales que requieren seguridad:

🏛️ Gobierno
Catastro municipal: datos de propiedades privadas NO deben ser públicos. Solo funcionarios autorizados.
🏗️ Empresa
Consultora ambiental: datos de clientes son confidenciales. Cada cliente solo ve sus propias capas.
🌐 IDE pública
Algunos datos son públicos (límites, ríos) pero otros son restringidos (infraestructura crítica).

🧩 Los 3 Conceptos de Seguridad

Usuarios → Roles → Reglas

La seguridad de GeoServer se basa en 3 niveles que trabajan juntos:

👤
1. Usuarios
Personas con credenciales (usuario + contraseña). Cada persona que accede tiene su propio usuario.
🏷️
2. Roles
Etiquetas que agrupan permisos. Un usuario puede tener uno o varios roles (ej: "editor", "visor").
📋
3. Reglas
Definen qué puede hacer cada rol: "el rol visor_publico puede leer la capa países pero NO la capa catastro".
Flujo de seguridad
Usuario "maria" (contraseña: *****)
  └── Tiene rol: "analista_ambiental"
       └── Regla: el rol "analista_ambiental" puede:
            ✅ Leer workspace "datos_ambientales"
            ✅ Leer workspace "datos_publicos"
            ❌ NO puede leer workspace "catastro"
            ❌ NO puede acceder a administración
💡
Piensa en roles, no en usuarios. No creas reglas para cada usuario individual. Crea roles con permisos y luego asigna usuarios a roles. Si mañana llega un nuevo analista, solo le asignas el rol "analista" y hereda todos los permisos.

🔑 Paso 1 — Cambiar la Contraseña de Admin

Lo primero que debes hacer en cualquier instalación

Menú lateral → Security → Users, Groups, and Roles → pestaña Users/Groups → clic en admin

Configuración
Password:         tu_nueva_contraseña_segura
Confirm password: tu_nueva_contraseña_segura
→ Save
⚠️
La contraseña por defecto geoserver es conocida por todo el mundo. Si tu servidor es accesible desde internet y no la cambias, cualquiera puede entrar y modificar o borrar tus datos. Cámbiala AHORA.

🏷️ Paso 2 — Crear Roles

Diseñar los roles antes de crear usuarios

Piensa primero qué tipos de acceso necesitas. Ejemplo para un proyecto típico:

ROLE_ADMINAcceso total. Administra GeoServer: crea capas, estilos, usuarios. Ya existe por defecto.
ROLE_VISORSolo puede VER capas públicas. No puede editar ni acceder a datos restringidos.
ROLE_EDITORPuede ver TODAS las capas y editar datos vía WFS-T. Para técnicos del equipo.
ROLE_PROYECTO_XSolo puede ver las capas del workspace "proyecto_x". Para clientes específicos.

Menú lateral → Security → Users, Groups, and Roles → pestaña Roles → Add new role

Crear cada rol
Role name: ROLE_VISOR
→ Save

Role name: ROLE_EDITOR
→ Save
💡
Convención de nombres: Los roles en GeoServer empiezan con ROLE_ por convención. No es obligatorio, pero es la práctica estándar y ayuda a identificarlos.

👤 Paso 3 — Crear Usuarios y Asignar Roles

Cada persona tiene su propio usuario

Menú lateral → Security → Users, Groups, and Roles → pestaña Users/Groups → Add new user

Ejemplo: Usuario visor público
Username:  visor_publico
Password:  contraseña_segura
Enabled:   ✓

Roles asignados: ROLE_VISOR
→ Seleccionar ROLE_VISOR en la lista y mover a "Selected"
→ Save
Ejemplo: Usuario técnico
Username:  tecnico_juan
Password:  contraseña_segura
Enabled:   ✓

Roles asignados: ROLE_EDITOR
→ Save
💡
Un usuario puede tener múltiples roles. Si Juan es técnico Y también necesita ver datos del proyecto X, le asignas ROLE_EDITOR y ROLE_PROYECTO_X. Los permisos se suman.

📋 Paso 4 — Reglas de Acceso a Datos

¿Qué puede ver cada rol?

Menú lateral → Security → Data → Add new rule

Cada regla define: Workspace + Layer + Access mode + Roles

Regla 1 — Todo el mundo puede leer datos públicos
Workspace:    mi_proyecto
Layer:        *  (todas las capas)
Access mode:  Read
Roles:        ROLE_VISOR, ROLE_EDITOR, ROLE_ADMIN
Regla 2 — Solo editores pueden escribir
Workspace:    mi_proyecto
Layer:        *
Access mode:  Write
Roles:        ROLE_EDITOR, ROLE_ADMIN
Regla 3 — Restringir una capa sensible
Workspace:    datos_privados
Layer:        catastro
Access mode:  Read
Roles:        ROLE_EDITOR, ROLE_ADMIN
→ ROLE_VISOR NO está en la lista = no puede ver esta capa
⚠️
Cuidado con el orden de las reglas. GeoServer evalúa las reglas de más específica a más general. Una regla para mi_proyecto:catastro tiene prioridad sobre una regla para mi_proyecto:*.
Access modes explicados
Read (r)Puede ver la capa vía WMS y descargar datos vía WFS. Es el permiso más básico.
Write (w)Puede crear, modificar y eliminar features vía WFS-T (transaccional). Para editores.
Admin (a)Puede administrar la capa: cambiar estilos, metadatos, configuración. Solo administradores.

🔗 Paso 5 — Reglas de Acceso a Servicios

Controlar qué servicios están disponibles

Menú lateral → Security → Services

Puedes restringir operaciones específicas. Por ejemplo: permitir WMS (ver mapas) pero bloquear WFS (descargar datos) para ciertos roles.

Ejemplo: Visor solo puede usar WMS, no WFS
Service:    WFS
Method:     *  (todas las operaciones)
Roles:      ROLE_EDITOR, ROLE_ADMIN
→ ROLE_VISOR no está = no puede usar WFS
→ El visor puede VER mapas (WMS) pero NO descargar datos (WFS)
💡
Caso real: Una IDE pública permite que cualquiera vea los mapas (WMS abierto) pero solo usuarios registrados pueden descargar datos (WFS restringido). Esto protege los datos de descargas masivas no autorizadas.

🧪 Paso 6 — Probar la Seguridad

Verificar que las restricciones funcionan

La mejor forma de probar es abrir una ventana de incógnito del navegador (sin la sesión de admin) y acceder como cada usuario:

Pruebas paso a paso
# 1. Abrir ventana de incógnito
# 2. Ir a: http://localhost:8080/geoserver/web/
# 3. Login como "visor_publico"
# 4. Verificar:
#    ✅ ¿Ve las capas públicas en Layer Preview?
#    ❌ ¿NO ve las capas restringidas?
#    ❌ ¿NO puede acceder a Styles, Stores, Workspaces?

# 5. Probar WMS directamente en la URL:
http://visor_publico:contraseña@localhost:8080/geoserver/mi_proyecto/wms?
  service=WMS&request=GetMap&layers=mi_proyecto:paises&...

# 6. Probar WFS (debe fallar para ROLE_VISOR si lo restringiste):
http://visor_publico:contraseña@localhost:8080/geoserver/mi_proyecto/wfs?
  service=WFS&request=GetFeature&typeName=mi_proyecto:paises&...
💡
Puedes pasar credenciales en la URL con el formato usuario:contraseña@host. Esto es útil para probar rápido, pero NUNCA lo uses en producción — las contraseñas quedan visibles en el historial.
Probar desde QGIS

Al agregar un servicio WMS/WFS en QGIS, hay campos de usuario y contraseña en la configuración de la conexión. QGIS envía las credenciales en cada petición.

Prueba: conecta como visor_publico y verifica que solo ve las capas permitidas. Luego conecta como tecnico_juan y verifica que ve más capas.

✨ Buenas Prácticas de Seguridad

1
Cambiar la contraseña de admin inmediatamente.

La contraseña por defecto es pública. Es lo primero que prueban los atacantes.

2
Diseñar roles antes de crear usuarios.

Piensa qué tipos de acceso necesitas y luego crea los roles. No hagas reglas por usuario individual.

3
Principio de mínimo privilegio.

Cada usuario solo debe tener acceso a lo que necesita. Si un visor no necesita WFS, no se lo des.

4
Organizar datos sensibles en workspaces separados.

Workspace "publico" para datos abiertos. Workspace "privado" para datos restringidos. Las reglas se aplican por workspace.

5
Probar SIEMPRE con una ventana de incógnito.

Tu sesión de admin tiene acceso a todo. La ventana de incógnito simula un usuario externo.

6
En producción: HTTPS obligatorio.

Sin HTTPS las contraseñas viajan en texto plano.

📖 Glosario — Clase 06

UsuarioPersona con credenciales (nombre + contraseña) que accede a GeoServer.
RolEtiqueta que agrupa permisos. Los usuarios se asignan a roles, los roles tienen reglas.
Regla de datosDefine qué rol puede leer/escribir qué workspace o capa.
Regla de servicioDefine qué rol puede usar qué servicio (WMS, WFS, WCS).
Read (r)Permiso de lectura: ver capas WMS, descargar datos WFS.
Write (w)Permiso de escritura: crear, editar y borrar features vía WFS-T.
Admin (a)Permiso de administración: cambiar configuración de capas, estilos, stores.
ROLE_ADMINRol de administrador que viene por defecto. Tiene acceso total.
Mínimo privilegioPrincipio: cada usuario solo tiene acceso a lo estrictamente necesario.
HTTPSHTTP seguro con cifrado. Obligatorio en producción para proteger contraseñas.
WFS-TWFS Transaccional. Permite crear, editar y borrar features remotamente.
💻 Práctica 06
Seguridad Completa
  1. Cambia la contraseña de admin a una contraseña segura (mínimo 10 caracteres).
  2. Crea 3 roles: ROLE_VISOR, ROLE_EDITOR, y un rol personalizado para un proyecto (ej: ROLE_PROYECTO_A).
  3. Crea 3 usuarios: uno para cada rol. Asigna los roles correspondientes.
  4. Crea reglas de datos: El visor puede leer el workspace principal. El editor puede leer y escribir. El usuario de proyecto solo puede leer un workspace específico.
  5. Restringe WFS: Crea una regla de servicio que impida al visor usar WFS (solo puede ver mapas con WMS).
  6. Prueba en ventana de incógnito: Login como visor → verifica que no ve capas restringidas. Login como editor → verifica que ve todo.
  7. Prueba desde QGIS: Conecta como cada usuario y verifica que ve solo lo que debe ver.
  8. Prueba GetMap con credenciales: Pega una URL WMS con usuario:contraseña@host y verifica el acceso.
🚀 Reto Extra — Escenario multicliente
Consultora con 2 clientes

Simula una consultora con 2 clientes: "Municipio Norte" y "Municipio Sur". Crea 2 workspaces, 2 roles (ROLE_NORTE, ROLE_SUR), y 2 usuarios. Cada cliente solo debe ver sus propias capas. Verifica cruzado: el usuario Norte NO debe ver datos del Sur.

  • Al menos 3 roles creados
  • Al menos 3 usuarios con roles asignados
  • Reglas de datos configuradas (read/write por workspace)
  • Regla de servicio WFS restringida para visor
  • Probado en ventana de incógnito: visor vs editor
  • Probado desde QGIS con credenciales
  • Entiendo el principio de mínimo privilegio
